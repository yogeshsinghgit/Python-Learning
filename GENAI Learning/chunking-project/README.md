# Chunking and Vector Ingestion Pipeline

## Overview

This project demonstrates the **Knowledge Ingestion Pipeline** used in Retrieval-Augmented Generation (RAG) systems.

The goal of this project is to:

1. Load documents from disk.
2. Split documents into meaningful chunks.
3. Generate embeddings for each chunk.
4. Store embeddings and metadata in Qdrant.

This project does **not** perform retrieval or LLM generation. It focuses only on the ingestion phase of a RAG system.

---

# Key Concepts

## Document

A source file that contains knowledge.

Examples:

* Markdown files
* PDFs
* Documentation
* Knowledge Base Articles

In this project, Markdown files are used as the source documents.

---

## Chunk

A smaller piece of a document.

Instead of generating one embedding for an entire document, the document is divided into multiple chunks.

Benefits:

* Better retrieval precision
* Improved semantic search
* Reduced context dilution
* Better scalability

---

## Embedding

An embedding is a numerical representation of text.

Example:

```text
JWT Authentication
       ↓
Embedding Model
       ↓
[0.12, -0.44, 0.88, ...]
```

Embeddings allow semantic similarity search.

---

## Vector

The output of an embedding model.

Each chunk is converted into a vector before being stored in Qdrant.

All vectors generated by the same embedding model have the same dimensionality.

Example:

```text
384 Dimensions
```

---

## Qdrant

Qdrant is a Vector Database.

It stores:

* Vectors
* Metadata (Payload)
* Similarity indexes

Qdrant allows efficient vector similarity search using algorithms such as HNSW.

# Services Overview

## DocumentLoader

### Responsibility

Loads documents from the local file system.

### Input

```text
docs/*.md
```

### Output

```text
Document Objects
```

### Why It Exists

Separates document loading logic from the rest of the application.

Future document sources could include:

* PDFs
* SharePoint
* Confluence
* Databases

without changing the ingestion workflow.

---

## ChunkingService

### Responsibility

Splits large documents into smaller chunks.

### Strategy Used

Recursive Chunking

### Why It Exists

Large documents produce poor embeddings.

Chunking improves:

* Precision
* Recall
* Retrieval quality

### Output

```text
Document
    ↓
Chunk 1
Chunk 2
Chunk 3
```

---

## EmbeddingService

### Responsibility

Converts chunks into vectors.

### Input

```text
Chunk Text
```

### Output

```text
Dense Vector
```

### Why It Exists

Vector databases cannot perform semantic search on raw text.

Embeddings transform text into a format that allows similarity comparison.

---

## QdrantRepository

### Responsibility

Handles all communication with Qdrant.

### Responsibilities Include

* Creating collections
* Inserting vectors
* Storing payload metadata

### Why It Exists

Separates database logic from business logic.

This allows switching to another vector database in the future with minimal changes.

---

## IngestionService

### Responsibility

Orchestrates the entire ingestion process.

### Workflow

```text
Load Documents
       ↓
Chunk Documents
       ↓
Generate Embeddings
       ↓
Store in Qdrant
```

This service acts as the central coordinator.

# Chunk Metadata

Each chunk stores metadata alongside its vector.

Example Metadata:

```text
Chunk ID
Document ID
Chunk Index
Chunk Content
```

Metadata allows:

* Filtering
* Traceability
* Source attribution
* Retrieval debugging

# Running Qdrant

## Start Qdrant

From the project root:

```bash
docker compose up -d
```

Verify the container is running:

```bash
docker ps
```

You should see the Qdrant container running.

# Accessing the Qdrant Dashboard

Open:

```text
http://localhost:6333/dashboard
```

The dashboard allows you to:

* View collections
* Inspect points
* View payload metadata
* Verify vector counts
* Debug ingestion

# Verifying Data in Qdrant

After running the ingestion pipeline:

1. Open the dashboard.
2. Open the collection.
3. Verify the number of points.
4. Inspect stored payload metadata.
5. Confirm vectors are present.

For the sample document used during development:

```text
Authentication.md
```

the collection contained:

```text
3 Points
```

corresponding to the generated chunks.

# End-to-End Flow

```text

        ↓

        ↓
Document Object
        ↓
ChunkingService
        ↓
3 Chunks
        ↓
EmbeddingService
        ↓
3 Vectors
        ↓
QdrantRepository
        ↓
Qdrant Collection
```

Each chunk becomes a separate vector point in Qdrant.
